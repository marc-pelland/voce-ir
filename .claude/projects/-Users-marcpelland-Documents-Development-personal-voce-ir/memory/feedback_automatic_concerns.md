---
name: Automatic Security, Testing, Documentation
description: Marc wants security/testing/docs to be automatic system concerns, not opt-in developer tasks. Users should understand what they built without understanding code.
type: feedback
---

Security, testing, and documentation should be automatic — generated by the system, not configured by the user. The principle: because Voce IR controls the entire pipeline, these concerns happen because the system was designed for them, not because a developer remembered to add them.

**Why:** Marc's vision is that this replaces systems like Next.js. Production apps need security, testing, and documentation — but in current tools these are afterthoughts. AI-generated code is especially bad at security (studies show LLMs frequently generate XSS-vulnerable code). Making these automatic is a competitive advantage.

**How to apply:**
- Always include security validation in the validator pipeline (OWASP mapping)
- Compiled output should include secure defaults (CSP, etc.) without configuration
- Every compilation generates a test report and application manifest
- The `voce` CLI should expose all of these as subcommands
- Style packs (design pattern libraries) should be the LoRA-equivalent for design quality
- Multi-agent AI architecture: different models for different roles (Opus for intent/design, Sonnet for generation, Haiku for repair)
