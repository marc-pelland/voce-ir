Specification Document — Draft v0.1

Voce IR

An intermediate representation for AI-generated user interfaces. Optimized for machine authorship, compiled for human experience. No human-readable code in the pipeline — just intent in, extraordinary UI out.

Marc Pelland Status: RFC 2026-03-20
01 — Thesis

Code is a constraint, not a feature

Every framework in use today — React, Vue, Svelte, Angular — was designed with one core assumption: a human will read and write this code. That assumption shapes everything: syntax must be learnable, abstractions must map to human mental models, error messages must be diagnostic, and the module structure must be navigable.

In a world where AI generates the vast majority of interface code, that assumption becomes a performance tax. Every abstraction that exists for human comprehension — the virtual DOM, CSS cascade, JSX transpilation, hook ordering rules — is overhead that the machine pays to communicate with itself in a language designed for someone else.

Voce IR removes the human from the code. The human stays in the loop — describing intent, reviewing output, guiding iteration — but never touches the intermediate representation. The IR is a binary, typed, machine-optimized format that an AI generates and a compiler consumes. It is not designed to be read. It is designed to be fast, correct, and expressive.

Design principle: The quality of the output is the only metric that matters. The IR exists to maximize the expressiveness, performance, and stability of compiled interfaces — not to be understood by a developer reading source files.

02 — Output Quality Pillars

Built for the human who sees it, not the one who codes it

The IR must be expressive enough to produce interfaces that meet three non-negotiable quality standards. These pillars constrain every design decision in the specification.

◆

Stability

Compiled output must be deterministic, crash-free, and resilient across devices and network conditions. The IR's type system and constraint validation make entire categories of runtime error structurally impossible.

◇

Experience

3D scenes, choreographed animation sequences, particle systems, shader effects, scroll-driven interactions, and personalized theming must be first-class primitives — not afterthoughts bolted on via third-party libraries.

○

Accessibility

Every visual node in the IR carries a parallel semantic node. Accessibility is not opt-in — it is structurally required by the format. The compiler enforces semantic completeness before emitting output.

Enforcement: The compiler will refuse to emit output if the IR fails validation against any pillar. A missing semantic label is a compile error, not a linting warning. A state machine with an unreachable state is rejected. An animation without a reduced-motion fallback is invalid.

03 — Architecture

Four-layer pipeline

Intent
The human interface. Natural language conversation with constraints: "Build a product detail page with a 3D model viewer, add-to-cart animation, and support for screen readers. Target: 120fps on M-series chips, graceful degradation to 2D on low-end Android." The AI parses intent, resolves ambiguity, and asks clarifying questions before emitting IR.
Generation
The AI engine. Translates intent into valid IR. The engine understands the IR schema natively — it does not generate human-readable code and then convert it. It emits typed binary IR directly, validating constraints as it generates. Generation can be incremental: a change to one view's state machine does not require regenerating the entire IR.
Compilation
The optimizing compiler. Takes IR + a device profile and emits the smallest, fastest possible output for the target. Whole-program optimization: dead state elimination, data request fusion, layout pre-computation, animation frame scheduling, shader compilation. The device profile includes GPU capabilities, screen size, refresh rate, network conditions, and accessibility preferences.
Runtime
The execution target. DOM (minimal, surgical mutations — no virtual DOM), WebGPU (direct draw calls for experiential/3D), WASM (near-native compute for complex state), Native (platform APIs for iOS/Android/desktop), or hybrid combinations where the compiler mixes targets per-component based on what each component needs.
04 — IR Node Types

The primitives

The IR is a directed acyclic graph of typed nodes. Each node has a fixed schema, typed inputs and outputs, and a set of constraints that must be satisfied for the IR to validate. Below are the core node types, grouped by domain.

Scene & Layout

Layout is expressed as a spatial graph with explicit constraints — not CSS. Every spatial relationship is declared, never inherited or cascaded. The compiler resolves constraints into concrete coordinates at build time when possible, or emits a minimal constraint solver for runtime-dependent layouts.

NodeDescriptionCompiles to
ViewRoot Top-level container. Defines viewport bounds, base theme, and the semantic document root. One per route. document / canvas / scene
Container Groups children with a layout strategy: stack (vertical/horizontal), grid (explicit rows/cols), absolute (positioned by anchor + offset), or flex (proportional allocation). div / flex / GPU quad
Surface A visible rectangular region with fill, stroke, corner radius, and optional clipping. The base visual primitive for 2D interfaces. div / rect / draw call
TextNode Styled text content. Properties: content (string or data binding), typeface, size, weight, color, line height, truncation strategy, locale. No CSS text properties — all explicit. span / text / glyph mesh
MediaNode Image, video, or audio with explicit dimensions, loading strategy (eager, lazy, placeholder), format negotiation, and alt description (required). img / video / texture
Scene3D A 3D scene subgraph. Contains camera, lights, meshes, materials, and an animation timeline. The compiler decides whether to emit WebGPU, Three.js, or a native 3D renderer. WebGPU / Three.js / SceneKit
MeshNode Geometry + material reference within a Scene3D. Supports instancing, LOD levels, and bounding volume hints for culling. Geometry can reference external glTF/USD assets. GPU mesh / instanced draw
ShaderNode Custom shader definition for materials or post-processing. Expressed as a portable shader IR (subset of WGSL) that the compiler translates to the target's shader language. WGSL / GLSL / Metal
ParticleSystem GPU-driven particle emitter with configurable spawn rate, velocity field, color over lifetime, and collision bounds. Used for ambient effects, transitions, and environmental detail. Compute shader / Canvas

State & Logic

All state is modeled as explicit, typed finite state machines. No implicit closures, no dependency arrays, no hook ordering. The compiler can statically analyze every possible state transition and eliminate unreachable paths.

NodeDescriptionCompiles to
StateMachine Named machine with typed states, typed transitions, guard conditions, and side effects. Supports hierarchical (nested) and parallel (concurrent) state machines. Every component's behavior is a state machine. Switch / lookup table
DataNode Declares an external data dependency with type, source (API, static, realtime), caching policy, refresh interval, and error/loading states. The compiler fuses co-located DataNodes into batched requests. fetch / WS / cache read
ComputeNode A pure function that transforms inputs to outputs. Used for derived state, formatting, filtering. Referentially transparent — the compiler can memoize or pre-compute at build time. Inline JS / WASM func
EffectNode A side effect triggered by a state transition: analytics event, API mutation, local storage write, haptic feedback. Effects are always attached to transitions, never to states — eliminating "effect on mount/unmount" ambiguity. Imperative call
ContextNode Shared state scoped to a subtree. Replaces React Context, Redux stores, and prop drilling. Typed, with explicit read/write boundaries — a child can read context without being able to trigger transitions on it. Scoped variable / closure

Motion & Interaction

Animation is a first-class IR concern, not a CSS afterthought. Every motion declaration includes a reduced-motion fallback (required by the validator). The compiler can schedule animations on the GPU compositor thread, pre-compute keyframe values, and batch transform operations.

NodeDescriptionCompiles to
Transition Animates property changes between state machine states. Properties: target node, property path, from/to values, duration, easing (cubic-bezier or spring physics), delay, and reduced-motion alternative. CSS transition / tween / GPU anim
Sequence Choreographed animation timeline: multiple Transitions played in sequence or parallel with stagger offsets. Supports labels for seeking and looping. Think GSAP timeline, but as IR data. Scheduled animation chain
GestureHandler Maps touch/mouse/keyboard input to state transitions or continuous property updates. Supports tap, drag, pinch, long-press, swipe with configurable thresholds and velocity curves. Keyboard equivalents are required. Event listener / pointer API
ScrollBinding Binds a node's properties to scroll position within a scrollable container. Properties: scroll range (start/end), property to animate, value mapping curve. Used for parallax, reveal effects, and progress indicators. Scroll observer / GPU scroll
PhysicsBody Attaches physics simulation to a node: mass, velocity, friction, restitution, constraints. Used for spring animations, momentum scrolling, draggable elements, and procedural motion. The compiler selects a solver appropriate for the target. Physics engine / spring fn

Navigation & Routing

NodeDescriptionCompiles to
RouteMap A state machine where states are views and transitions are navigation events. Supports nested routes (layouts wrapping child routes), guards (auth checks), and data preloading tied to route entry. History API / deep links
RouteTransition Defines the visual transition between routes: shared element morphing, page-level enter/exit animations, and loading state handling. The compiler ensures smooth handoff between outgoing and incoming views. View Transitions API / custom

Accessibility & Semantics

NodeDescriptionCompiles to
SemanticNode Parallel tree mirroring every visual node. Carries: role (button, heading, list, landmark, etc.), label (human-readable text), relationships (labelledby, describedby, controls), and keyboard focus order. Required. The compiler rejects IR where any interactive visual node lacks a SemanticNode. ARIA attrs / a11y API
LiveRegion Declares a region whose content changes should be announced by screen readers. Properties: politeness (polite, assertive), atomic (announce whole region or just changes), relevant (additions, removals, text). aria-live region
FocusTrap Constrains keyboard focus to a subtree (for modals, drawers, dialogs). Specifies initial focus target, tab order within the trap, and escape behavior. Focus management JS
ReducedMotion Every Transition and Sequence node must reference a ReducedMotion alternative. This can be: a simplified transition (opacity fade instead of complex movement), an instant cut, or a static state. The compiler enforces this at validation time. prefers-reduced-motion

Theming & Personalization

NodeDescriptionCompiles to
ThemeNode A named set of design tokens: colors, typefaces, spacing scale, border radii, shadow definitions. Multiple themes can coexist (light, dark, high-contrast, brand variants). Theme selection is a state machine transition. CSS vars / runtime tokens
PersonalizationSlot A point in the IR that adapts based on user context: locale, preferences, device capabilities, A/B test cohort, or external personalization API. The slot declares its variants and the conditions under which each activates. Conditional branch
ResponsiveRule Adapts layout and content based on viewport dimensions, device pixel ratio, or input modality (touch vs pointer vs keyboard). Expressed as explicit breakpoints with full layout overrides — not CSS media queries with cascading patches. Container query / layout swap
05 — Type System

Correctness by construction

The IR's type system is structural and strict. Every value has a known type at IR generation time. The compiler performs full type checking before emitting output — there are no runtime type errors in compiled Voce applications.

Primitive types

Int32, Int64, Float32, Float64     // Numeric
Bool                                // Boolean
String                              // UTF-8 text
Color                               // RGBA, supports color spaces
Length                               // px | rem | % | vw | vh | auto
Duration                            // ms, with min/max constraints
Angle                               // deg | rad
Vec2, Vec3, Vec4                    // Spatial vectors
Mat3x3, Mat4x4                     // Transform matrices
Quaternion                          // 3D rotation
Easing                              // cubic-bezier(a,b,c,d) | spring(m,s,d)

Composite types

Struct { field: Type, ... }         // Named product type
Enum { Variant1, Variant2(Type) }   // Tagged union
List<T>                             // Ordered collection
Map<K, V>                           // Key-value store
Optional<T>                         // Nullable value
Result<T, E>                        // Success or typed error
Asset<mime>                         // External resource reference
Ref<NodeType>                       // Cross-reference to another IR node

Constraint types

These are the mechanism by which the IR enforces the three pillars at the type level:

Bounded<T, min, max>                // Value within range (stability)
NonEmpty<String>                     // Must have content (accessibility)
Accessible<T>                       // T must have SemanticNode (accessibility)
MotionSafe<T>                       // T must have ReducedMotion alt (accessibility)
Cacheable<T, ttl>                   // Value can be cached for ttl (performance)

Key insight: The constraint types encode the output quality pillars directly into the type system. A GestureHandler that lacks a keyboard equivalent won't type-check as Accessible. An animation without a ReducedMotion fallback won't type-check as MotionSafe. The compiler catches pillar violations at compile time, not at runtime or in QA.

06 — Compiler

Target-aware optimization

The compiler accepts the validated IR plus a device profile and emits the minimal, maximally optimized output for that target. The device profile is a structured description of the execution environment.

Device profile fields

DeviceProfile {
  target: DOM | WebGPU | WASM | Native(platform)
  gpu: { vendor, vram_mb, compute_shaders: bool, max_texture_size }
  display: { width, height, dpr, refresh_hz, hdr: bool, color_gamut }
  input: { touch: bool, pointer: bool, keyboard: bool, gamepad: bool }
  network: { type: wifi | cellular | offline, bandwidth_mbps }
  a11y: { screen_reader: bool, reduced_motion: bool, high_contrast: bool }
  performance_budget: { js_kb, ttfi_ms, memory_mb }
}

Optimization passes

  • Dead state elimination: States unreachable from any route entry point are removed. Effects attached to dead transitions are stripped.
  • Data fusion: Multiple DataNodes fetching from the same origin are merged into a single batched request. Co-dependent data flows are resolved into an optimal fetch graph.
  • Layout pre-computation: Containers whose dimensions don't depend on runtime data are resolved to static coordinates at compile time. Only runtime-dependent layouts emit constraint-solving code.
  • Animation scheduling: Transitions are analyzed for GPU composability. Compositor-safe properties (transform, opacity) are separated from layout-triggering properties and scheduled on the GPU thread.
  • Shader compilation: ShaderNodes are compiled to the target's shader language (WGSL for WebGPU, GLSL for WebGL fallback, Metal for iOS). Unused shader variants are stripped.
  • Asset optimization: Images are resized and format-negotiated for the target display (AVIF for capable browsers, WebP fallback). 3D assets are LOD-selected based on GPU capability. Fonts are subset to used glyphs only.
  • Tree shaking: Nodes referenced only by dead routes or eliminated personalization variants are removed from the output.
  • A11y enforcement: Final pass verifies every interactive node has a semantic counterpart, every animation has a reduced-motion alternative, and keyboard navigation covers all interactive paths. Failure is a compile error.

07 — Experience Primitives

Extraordinary output is a specification requirement

Most frameworks treat 3D, complex animation, and environmental effects as external concerns — "add Three.js," "use GSAP," "install a particle library." Voce IR makes these first-class because the experiences humans love most are the ones current frameworks make hardest to build.

What the IR enables natively

  • 3D product viewers with orbit controls, environment lighting, and material switching — as a single Scene3D node, not a React-Three-Fiber dependency tree
  • Choreographed page transitions where shared elements morph across routes, background layers parallax, and content staggers in with spring physics — as a RouteTransition + Sequence, not 200 lines of GSAP
  • Scroll-driven storytelling where sections reveal, graphics animate, and data visualizations build as the user scrolls — as ScrollBinding nodes, not Intersection Observer boilerplate
  • Ambient particle effects that react to cursor position and device orientation — as a ParticleSystem with input bindings, not a custom WebGL pipeline
  • Physics-driven UI where dragged elements have momentum, snapping has spring dynamics, and dismissed cards fly off-screen with realistic velocity — as PhysicsBody nodes, not manual requestAnimationFrame loops
  • Personalized theming where colors, typography, layout density, and content adapt to user preferences, locale, and device — as PersonalizationSlot + ThemeNode, not a tangle of context providers and media queries

The key difference: In today's ecosystem, adding 3D or complex animation means adding dependencies, increasing bundle size, and hoping the libraries play well together. In Voce, these are compiler-optimized primitives. The compiler knows how to schedule a particle system on the GPU, how to batch mesh draw calls, and how to degrade gracefully when the target device can't handle it — all without the developer (or the AI) managing the plumbing.

08 — Conversational Building

The input is a conversation, the output is an experience

The AI generation layer is not a code-completion tool. It is the primary authoring interface. The quality of conversational building determines whether Voce IR can match the iterative flexibility of working in React with hot reload.

Intent resolution protocol

When a human describes an interface, the AI follows a structured resolution process before generating IR:

  1. Clarify scope: Is this a new view, a modification to an existing view, or a global change (theme, routing, data source)?
  2. Resolve ambiguity: "Make it feel premium" maps to specific IR properties — spring-physics easing, generous spacing, serif typography, subtle parallax. The AI proposes concrete interpretations and confirms.
  3. Check constraints: Does the request conflict with existing state machines, accessibility requirements, or performance budgets? If so, surface the conflict and propose alternatives.
  4. Generate incrementally: Emit only the IR nodes that changed. The compiler's incremental mode recompiles only affected subtrees.
  5. Preview and iterate: The compiled output renders instantly in a live preview. The human describes adjustments; the AI patches the IR. No save, no build, no refresh.

Example conversation flow

Human: "I want a landing page for a sneaker brand. Dark, bold,
        the shoe should float in 3D and react to mouse movement.
        Big typography. Add-to-cart should feel satisfying."

AI:    Generating ViewRoot with dark ThemeNode...
       Creating Scene3D with orbit-locked camera, mouse-reactive
       rotation via GestureHandler → continuous property update...
       TextNode with display typeface at 72px, spring-entrance
       Sequence on scroll...
       Add-to-cart button with haptic EffectNode on press,
       particle burst ParticleSystem on success transition,
       cart count LiveRegion for screen reader announcement...

       [Live preview renders]
Human: "The shoe rotation feels too fast, and I want the
        add-to-cart particle burst to be gold, not white."

AI:    Patching GestureHandler rotation sensitivity from 1.0 to 0.4...
       Updating ParticleSystem color_over_lifetime to gold ramp...
       [Preview updates in ~200ms]

The conversational loop replaces the code-save-reload loop. The human never leaves the conversation to edit a file. The AI never generates code the human needs to read. The preview is the source of truth — if it looks right and passes validation, it is right.

09 — Visual Inspector

Debugging without code

If humans never see the IR, how do they debug? The answer is a visual inspector that exposes the runtime behavior of the compiled application in human-comprehensible terms — without showing a single line of source code.

Inspector panels

  • Scene graph overlay: Click any element in the live preview to see its IR node type, current properties, data bindings, and position in the layout graph. Similar to browser DevTools element inspector, but showing IR-level information instead of HTML/CSS.
  • State machine visualizer: Live diagram of every active state machine, current state highlighted, available transitions shown as edges. Click a transition to see its guard conditions and effects. Trace history to see how the application reached its current state.
  • Data flow monitor: Shows all active DataNodes, their fetch status (loading, cached, stale, error), response timing, and which visual nodes depend on them. Highlights data waterfalls and suggests fusion opportunities.
  • Animation timeline: Scrubable timeline of all active Sequences and Transitions. Pause, step frame-by-frame, adjust easing curves live. Shows GPU vs main thread scheduling.
  • Accessibility audit: Live a11y tree alongside the visual tree. Tab through focus order, see screen reader announcements in real time, verify contrast ratios, test reduced-motion alternatives.
  • Performance profiler: Frame-by-frame breakdown. GPU utilization, layout computation time, memory allocation. Flags any frame that exceeds the performance budget defined in the device profile.

The inspector is conversational too. Instead of manually tracing through inspector panels, the human can describe the bug: "The modal isn't dismissing when I tap outside it." The AI traces the relevant state machine, identifies that the GestureHandler's tap-outside detection region doesn't extend to the overlay area, and patches the IR. The inspector is a fallback for when you want to explore — the conversation is the primary debugging interface.

10 — Implementation Roadmap

From specification to system

Phase 1
IR specification & validator. Formal binary format definition using FlatBuffers or similar. Schema for all node types. Reference validator that accepts/rejects IR blobs with typed error messages. Accessibility and stability constraints enforced. Target: 8-12 weeks.
Phase 2
DOM compiler (MVP target). Takes validated IR and emits minimal HTML + JS. No framework runtime. Surgical DOM mutations. Inline styles. Direct event binding. Proves the concept compiles and runs. Target: 12-16 weeks.
Phase 3
AI generation layer. Prompt-engineered or fine-tuned model that emits valid IR from natural language. Iterative: discover missing primitives, expand IR spec, retrain. Conversational loop with live preview. Target: 16-24 weeks (ongoing iteration).
Phase 4
WebGPU + WASM compile targets. 3D scenes compile to WebGPU draw calls. Compute-heavy state machines compile to WASM. Hybrid mode where the compiler mixes DOM and WebGPU per-component. Target: 12-16 weeks.
Phase 5
Visual inspector + feedback tooling. Scene graph overlay, state machine visualizer, animation timeline, a11y auditor, performance profiler. Conversational debugging integration. Target: 12-16 weeks.
Phase 6
Native compile targets + ecosystem. iOS (SwiftUI/Metal), Android (Compose/Vulkan), desktop, and purpose-built runtimes (kiosk, TouchDesigner bridge). Plugin ecosystem for custom node types. Open-source IR spec for community adoption. Target: ongoing.
11 — Open Questions

Unresolved design decisions

  • IR versioning: How does the format evolve without breaking existing compiled artifacts? Semantic versioning at the schema level? Forward-compatible field additions?
  • Escape hatches: When the IR cannot express a requirement, what is the fallback? Inline raw target code (loses portability)? An extension node type? A plugin system?
  • Collaboration model: How do multiple humans (or multiple AI agents) work on the same IR graph concurrently? CRDT-based merging? Intent-level conflict resolution?
  • Testing: How do you write tests for an interface with no source code? State machine assertions? Visual regression against reference renders? AI-generated test scenarios from the intent history?
  • Source control: Binary IR is not diffable in Git. Do we maintain a parallel human-readable serialization for version control, or build custom diff tooling?
  • Intellectual property: If the IR is generated by an AI and compiled by a toolchain, who owns the output? How does licensing work for the IR format itself?
  • Progressive enhancement: How does a Voce-compiled site degrade for a browser with JavaScript disabled? Is there a static HTML fallback layer in the IR?
